## 📚 Theory

### What feature engineering is

Transforming raw data into features that "highlight" the task for the model. Kinds:

1. **Aggregates over history:** sums, means, counts, shares over time windows (30/90/180 days).
2. **Ratios and differences:** payment / income, days since last delinquency.
3. **Temporal features:** day of week, account age, seasonality.
4. **Categorical encoding** — turning strings into numbers.
5. **Transformations:** log for skewed amounts, age binning.

### Encoding categorical features

| Method | How | When |
| --- | --- | --- |
| **One-Hot** | a column per category | few categories (< 15–20) |
| **Ordinal** | category → number | categories truly ordered (education) |
| **Target encoding** | category → mean target within it | many categories; ⚠️ leakage risk — compute on train only, with CV/smoothing |
| **Frequency** | category → its frequency | fast and often surprisingly good |

⚠️ Trap: one-hot for "city" with 900 values → 900 columns, a sparse matrix, overfitting to rare cities.

### Missing values: think like a production engineer

- Fill with statistics computed **on train** (otherwise leakage).
- A **missingness flag** as a separate column is often useful: "didn't state income" is itself a risk signal.
- In production a category/value unseen in training will arrive — the pipeline must survive it (`handle_unknown="ignore"`, defaults).

### Feature leakage (the main enemy of scoring)

A feature must not contain information **from the future relative to the application moment**. A classic failure: the feature "number of debt collector calls" — it only appears for those already delinquent. A model with it shows a fantastic AUC on history and collapses in production. The rule: **all features are computed strictly on data available before the application moment (point-in-time).**

## 💻 Practice

```python
import pandas as pd

apps = pd.read_csv("applications.csv", parse_dates=["app_date"])       # applications
txn  = pd.read_csv("transactions.csv", parse_dates=["txn_date"])       # wallet transactions
loans = pd.read_csv("past_loans.csv", parse_dates=["issue_date"])      # past installments

# --- 1. Transaction aggregates over the 90 days BEFORE the application (point-in-time!)
merged = txn.merge(apps[["user_id", "app_date"]], on="user_id")
merged = merged[(merged.txn_date < merged.app_date) &
                (merged.txn_date >= merged.app_date - pd.Timedelta(days=90))]

agg = merged.groupby("user_id").agg(
    txn_count_90d=("amount", "size"),
    txn_sum_90d=("amount", "sum"),
    txn_mean_90d=("amount", "mean"),
    inflow_90d=("amount", lambda s: s[s > 0].sum()),
    outflow_90d=("amount", lambda s: -s[s < 0].sum()),
)
agg["outflow_to_inflow"] = agg.outflow_90d / agg.inflow_90d.clip(lower=1)

# --- 2. Installment history
loan_agg = loans.groupby("user_id").agg(
    loans_count=("loan_id", "size"),
    had_delinquency=("max_delay_days", lambda s: (s > 30).any().astype(int)),
    avg_delay=("max_delay_days", "mean"),
)

# --- 3. Assembling the matrix
X = apps.merge(agg, on="user_id", how="left").merge(loan_agg, on="user_id", how="left")
X["is_new_user"] = X.txn_count_90d.isna().astype(int)   # the "no history" flag is itself a feature!
X["loans_count"] = X.loans_count.fillna(0)
X["requested_to_income"] = X.requested_amount / X.declared_income.clip(lower=1)
```

### Verifying feature value

```python
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.inspection import permutation_importance
# train and inspect permutation importance on validation:
# requested_to_income   0.081   ← the amount-to-income ratio is the top feature
# had_delinquency       0.064
# outflow_to_inflow     0.041
# is_new_user           0.037   ← the missingness flag works!
```

<details>
<summary><b>Igor's review</b></summary>

1. **The point-in-time filter is the main thing I check.** The `txn_date < app_date` window is mandatory; without it — future leakage and a paper "AUC 0.99".
2. `how="left"` in the merge: new users without history must not fall out of the sample — they are the riskiest.
3. `clip(lower=1)` in denominators — division by zero will crash the service in production.
4. The "requested/income" ratio is a typical example: two weak features apart, together — the top one. Inventing such ratios is the craft.
</details>

## 📌 Remember

- Scoring quality is won with features, not with model choice.
- Aggregates over time windows + ratios are the workhorses of tabular ML.
- Every feature is strictly point-in-time relative to the prediction moment.
- Missingness is information too; encode it as a flag.

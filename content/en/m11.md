3. Choose the threshold from economics and operational limits.

## 📚 Theory

### What breaks at 1:1000

1. **Accuracy is meaningless** (module 6): the constant "not fraud" gives 99.9%.
2. **The loss function barely sees the rare class:** 0.1% of examples contribute 0.1% of the gradient — the model learns to ignore fraud.
3. **ROC-AUC is deceptively high**; look at PR-AUC and precision/recall at the operating point.

### Ways to fight it

| Method | Essence | Pros / cons |
| --- | --- | --- |
| **Class weights** (`class_weight="balanced"`, `scale_pos_weight`) | errors on the rare class cost more | simple, no data distortion; the first choice |
| **Undersampling** | drop part of the majority class | faster training; loses information |
| **Oversampling** | duplicate the rare class | overfitting risk on duplicates |
| **SMOTE** | synthetic examples between rare-class neighbors | helps weak models; often useless for boosting, creates unrealistic points |
| **Threshold tuning** | don't touch the data, shift the decision | mandatory always, whatever you do above |

Industry practice: **for boosting — class weights + threshold tuning.** Sampling — when the data doesn't fit in memory or the model is very weak.

⚠️ **Golden rules:**
- Sampling is applied **to train only**, never to validation/test — otherwise the metrics are measured in a "fantasy" world.
- After resampling/weights the predicted probabilities are **biased** — they cannot be read as real frequencies without calibration (`CalibratedClassifierCV`).

### The threshold from economics

The expected value of an alert at threshold t:
`V(t) = TP(t)·220 − FP(t)·5`, subject to `TP(t)+FP(t) ≤ 500/day`.
The threshold is the solution of an optimization problem, not the constant 0.5.

## 💻 Practice

```python
from lightgbm import LGBMClassifier
from sklearn.metrics import precision_recall_curve, average_precision_score
import numpy as np

# imbalance: 0.1% fraud
pos_weight = (y_train == 0).sum() / (y_train == 1).sum()   # ≈ 999

model = LGBMClassifier(n_estimators=2000, learning_rate=0.05,
                       scale_pos_weight=pos_weight, random_state=42)
model.fit(X_train, y_train, eval_set=[(X_val, y_val)], eval_metric="average_precision",
          callbacks=[lightgbm.early_stopping(100)])

proba = model.predict_proba(X_test)[:, 1]
print(f"PR-AUC: {average_precision_score(y_test, proba):.3f}")   # 0.412
# (ROC-AUC is 0.972 at the same time — see how much it flatters?)
```

### The operating point: 500 alerts a day

```python
# test = 30 days → budget 500*30 = 15000 alerts
threshold = np.sort(proba)[-15000]           # threshold = the score of the 15000th transaction
alerts = proba >= threshold

tp = ((alerts) & (y_test == 1)).sum()        # 1113 caught
fp = ((alerts) & (y_test == 0)).sum()        # 13887 false
recall = tp / (y_test == 1).sum()            # 0.62 — we catch 62% of fraud
value = tp * 220 - fp * 5                    # ≈ +$175K per month
```

Strategy comparison at the same budget: no weights — recall 0.44; with weights — 0.62; SMOTE — 0.57 (worse than weights: synthetics blurred the boundary).

<summary><b>Review with Igor</b></summary>

1. **PR-AUC 0.41 vs ROC-AUC 0.97** — the typical imbalance picture. In the executive report we show recall and money at the operating point, not AUC.
2. **Threshold = the quantile matching the operational limit** (500 alerts/day) — the model is built into the review team's real process.
3. Precision at the operating point is just 7% — "93% of alerts are false, is that normal?!" Yes: an FN costs ~45× an FP; the economics work out. Always compute value, not percentages.
4. The next maturity step: **probability calibration** and prioritizing alerts by expected damage `P(fraud)·amount`, not by probability alone.
</details>

## 📌 Remember

- Under imbalance: PR-AUC + operating-point metrics; accuracy and "bare" ROC-AUC mislead.
- The first tool is class weights; sampling — train only; probabilities afterwards are biased.
- The threshold is derived from error costs and the team's throughput.
- Report to the business in money.

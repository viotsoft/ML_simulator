## 📚 Theory

### Maturity levels of ML systems

0. A notebook + manual runs ("us, today").
1. Model as a service: API, container, CI/CD, versions.
2. An automated pipeline: retraining on schedule/trigger, validation gates.
3. Full MLOps: feature store, model registry, automatic rollback, model A/B.

Most teams only need levels 1–2. Don't build level 3 for the badge.

### The model as an artifact

- **Serialization:** joblib/pickle (⚠️ library versions must match!), ONNX — a portable format.
- **Versioning:** the model + code + data + metrics are pinned together (MLflow Model Registry, W&B). The question "which model is in production and what was it trained on?" must be answerable in 10 seconds.
- **Reproducibility:** pinned random_state, package versions (a lock file), code in git.

### The three layers of ML service monitoring

1. **Service (like any backend):** p50/p99 latency, RPS, 5xx errors.
2. **Data drift:** the input feature distribution has drifted from training (a new customer segment, a broken field in a source). Metrics: PSI, KS test per feature. **Prediction drift** — a shift in the score distribution — is the cheapest early signal.
3. **Quality (concept drift):** true labels arrive late (we learn about churn 30 days later) → compute delayed quality, compare with the offline estimate.

The rule: **a model without monitoring is an incident that hasn't happened yet.** Data always changes.

### Model rollout strategies

- **Shadow mode:** the new model predicts in parallel; answers are logged but not used. A free check on real traffic.
- **Canary:** 5% of traffic to the new model; metrics fine → roll out.
- **Rollback:** the previous version always on standby; switching by config, not by deployment.

## 💻 Practice

### The FastAPI service

```python
# app.py
from fastapi import FastAPI
from pydantic import BaseModel, Field
import joblib

app = FastAPI(title="churn-model", version="2.3.0")
model = joblib.load("model/churn_lgbm_v2.3.joblib")

class Features(BaseModel):
    tenure_months: float = Field(ge=0)
    monthly_fee: float
    logins_last30: int = Field(ge=0)
    support_tickets: int
    negative_tickets: int
    days_since_last_login: float
    autopay_enabled: int

@app.post("/predict")
def predict(f: Features):
    proba = model.predict_proba([[*f.model_dump().values()]])[0, 1]
    return {"churn_probability": round(float(proba), 4),
            "model_version": "2.3.0"}

@app.get("/health")
def health():
    return {"status": "ok"}
```

Pydantic validates the input: a negative `tenure_months` returns a 422, not a garbage prediction.

### Dockerfile

```dockerfile
FROM python:3.12-slim
WORKDIR /srv
COPY requirements.lock .
RUN pip install --no-cache-dir -r requirements.lock
COPY app.py model/ ./
EXPOSE 8000
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
```

`requirements.lock` with exact versions — otherwise in six months the image builds with a different sklearn and the model silently changes its predictions.

### Drift monitoring (PSI)

```python
def psi(expected, actual, bins=10):
    """Population Stability Index: <0.1 fine, 0.1–0.25 watch, >0.25 alert"""
    cuts = np.quantile(expected, np.linspace(0, 1, bins + 1))
    e = np.histogram(expected, cuts)[0] / len(expected)
    a = np.histogram(actual,  cuts)[0] / len(actual)
    e, a = np.clip(e, 1e-4, None), np.clip(a, 1e-4, None)
    return np.sum((a - e) * np.log(a / e))

# a weekly job: PSI for every feature and for the model scores
# weeks 1–6: all PSI < 0.08
# week 7:  days_since_last_login PSI = 0.31  ← ALERT
```

Investigating the alert: the mobile app shipped auto-login — "days since last login" massively went to zero. The feature degraded → a scheduled retrain with a new activity definition. **The drift was caught before retention noticed the list quality dropping.**

1. `/health` for the orchestrator, the model version in every response — during an incident you instantly see who answered.
2. Log **the input and output** of every prediction (this is the future train set and the incident analysis material).
3. Shadow mode for a week before switching traffic.
4. A runbook: "model down → fallback to the rule tenure<3 & logins=0; rollback via the MODEL_VERSION env var".

## 📌 Remember

- A model in production = code + artifact + data + versions pinned together.
- Input validation (Pydantic) is cheap and saves you from garbage predictions.
- Monitoring: service metrics + drift (PSI on features and scores) + delayed quality.
- Shadow → canary → rollout; rollback by config in a minute.

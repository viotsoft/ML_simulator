## 📚 Theory

### Overfitting and underfitting

- **Underfitting (high bias):** the model is too simple; bad on both train and test.
- **Overfitting (high variance):** the model learned the noise; brilliant on train, a flop on new data.

Diagnosis by the gap: `train_metric` ≫ `val_metric` → overfitting. Both low → underfitting. Andrew Ng: "Look at train error and val error like a doctor at two lab tests — the combination gives the diagnosis."

Overfitting cures: more data, a simpler model, regularization, early stopping. Underfitting cures: a more complex model, better features, longer training.

### Cross-validation

One train/test split is one random estimate. **K-fold CV:** split the data into K parts, train K times on K−1 parts, validating on the remaining one; average. You get both the estimate and its spread.

- `StratifiedKFold` — under class imbalance.
- **`TimeSeriesSplit` / time-based split** — whenever data has time (almost always in business!). Train on the past, validate on the future. A random split of two years of transactions = the model "saw the future".
- **GroupKFold** — when one customer has many rows: all of a customer's rows must land in one fold, otherwise the model recognizes the customer, not the pattern.

### The data leakage catalog (learn by heart)

1. **Target leakage into features:** a feature is a consequence of the target ("number of collector calls" for default).
2. **Leakage through time:** features/aggregates computed using the future (not point-in-time).
3. **Leakage through preprocessing:** scaler/encoder/imputer fit on all data before the split.
4. **Leakage through duplicates/groups:** the same customer in both train and test.
5. **Leakage through feature selection:** selection done on all data, then an "honest" CV.

The symptom of all leaks is the same: **a suspiciously high offline metric and a production flop.**

## 💻 Practice: the investigation

You study the failed model's notebook.

```python
# CLUE 1: a random split of temporal data
X_train, X_test = train_test_split(df, test_size=0.2, random_state=42)
# two years of applications shuffled: trained on December, tested on January of the same year

# CLUE 2: preprocessing before the split
scaler = StandardScaler().fit(X)          # ← on ALL the data
X_scaled = scaler.transform(X)
X_train, X_test = train_test_split(X_scaled, ...)

# CLUE 3 (the main one): a feature from the future
features = [..., "account_status_at_export"]
# the account status AT EXPORT TIME, not at application time.
# Defaulters have status "blocked" — the model is simply reading the answer.
```

Testing the leakage hypothesis:

```python
from sklearn.model_selection import cross_val_score
# honest validation: time split + without the leaky feature
auc_leaky   = 0.97   # as they had
auc_no_leak = cross_val_score(model, X.drop(columns=["account_status_at_export"]),
                              y, cv=TimeSeriesSplit(5), scoring="roc_auc").mean()
print(auc_no_leak)   # 0.74 ← the model's real quality
```

A permutation test for leakage: a single feature carries importance 0.61 of 1.0 — one feature "explains" everything. A feature that is too good is almost always a leak.

<details>
<summary><b>The incident report (post-mortem)</b></summary>

**Cause:** the feature `account_status_at_export` contains information from the future (the status is set after the default). Additionally: a random split of temporal data and preprocessing before the split inflated the estimate.

**The model's real quality:** ROC-AUC 0.74 (not 0.97). The model went to production "blind": in real time the tell-tale feature always equals "active".

**Fixes (process-level, not only technical):**
1. Time-based validation is the standard for all models with temporal nature.
2. All preprocessing lives inside a `Pipeline` only.
3. Review check: "is every feature available at prediction time?" — a mandatory item.
4. Alert: if a single feature carries > 40% importance — manual leakage check.

Victor: "Great post-mortem. Note: we punish no one — we fix the process."
</details>

## 📌 Remember

- The train/val metric gap is the main overfitting indicator.
- Temporal data is validated by time; a customer's rows are never separated across folds or train/test.
- A metric that is too good is a reason for alarm, not joy.
- Preprocessing only inside a Pipeline; features only point-in-time.
